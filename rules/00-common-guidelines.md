# 공통 작업 지침 (모든 프로세스 공통 적용)

## 1. TASK 단위 작업 원칙

### 1.1 TASK 정의
- 모든 작업은 **TASK 단위로 분리**하여 진행해야 합니다
- 각 TASK는 **명확한 목표**와 **완료 기준**을 가져야 합니다
- TASK는 **독립적으로 실행 가능**하고 **검증 가능**해야 합니다
- TASK는 **하나의 파일 또는 관련된 파일 그룹**에 집중해야 합니다

### 1.2 TASK 분리 기준
- **기능 단위**: 하나의 기능 = 하나의 TASK
- **파일 단위**: 하나의 파일 수정 = 하나의 TASK
- **컴포넌트 단위**: 하나의 컴포넌트 개발/수정 = 하나의 TASK
- **API 단위**: 하나의 API 엔드포인트 = 하나의 TASK

### 1.3 TASK 우선순위
- **P0 (긴급)**: 시스템 동작에 필수적인 작업
- **P1 (높음)**: 핵심 기능 구현
- **P2 (보통)**: 일반 기능 구현
- **P3 (낮음)**: 개선 작업

## 2. 작업 수행 프로세스

### 2.1 Phase 1: 작업 계획 수립
1. **입력 데이터 분석**: 제공된 입력 데이터를 명확히 이해
2. **TASK 분해**: 큰 작업을 작은 TASK로 분해, 식별 가능한 최소한의 크기로 분해
3. **의존성 파악**: TASK 간 의존성 확인
4. **우선순위 결정**: TASK 실행 순서 결정
5. **작업 단계 계획**: 각 TASK별 작업 단계 및 예상 시간 계획

### 2.2 Phase 2: 작업 수행
1. **TASK 선택**: 우선순위에 따라 TASK 선택
2. **파일 확인**: 관련 파일 읽기 및 이해
3. **구현**: TASK 목표에 맞게 코드 작성/수정
4. **검증**: TASK 완료 기준 확인

### 2.3 Phase 3: 검증 및 완료
1. **로컬 테스트**: 변경 사항이 정상 동작하는지 확인
2. **린터 검사**: 코드 스타일 및 오류 확인
3. **TASK 완료**: 완료 기준 충족 시 TASK 완료 처리
4. **다음 TASK**: 다음 우선순위 TASK로 진행

## 3. TASK 작성 형식

### 3.1 TASK 템플릿
```markdown
## TASK: [작업명]
- **목표**: [명확한 목표 설명]
- **범위**: [수정/생성할 파일 목록]
- **완료 기준**: [완료 확인 방법]
- **우선순위**: P0/P1/P2/P3
- **의존성**: [선행 TASK 목록]
```

### 3.2 TASK 진행 체크리스트
각 TASK 진행 시 다음 항목을 확인:
- [ ] TASK 목표가 명확한가?
- [ ] 관련 파일을 모두 확인했는가?
- [ ] 의존성 TASK가 완료되었는가?
- [ ] 코드 작성/수정이 완료되었는가?
- [ ] 린터 오류가 없는가?
- [ ] 완료 기준을 충족했는가?

## 4. 작업 수행 시 준수 사항

### 4.1 필수 사항
1. **한 번에 하나의 TASK만 처리**: 여러 TASK를 동시에 진행하지 않음
2. **TASK 완료 후 검증**: 각 TASK 완료 시 반드시 검증 수행
3. **파일 단위 작업**: 하나의 파일 수정이 완료되면 다음 파일로 진행
4. **의존성 준수**: 의존성이 있는 TASK는 순서대로 진행

### 4.2 금지 사항
1. **여러 파일 동시 수정 금지**: 관련 없는 파일을 동시에 수정하지 않음
2. **검증 없이 다음 TASK 진행 금지**: TASK 완료 검증 없이 다음 작업 진행 금지
3. **의존성 무시 금지**: 의존성 관계를 무시하고 TASK 진행 금지

## 5. 결과 파일 저장 규칙

### 5.1 파일 저장 위치
- 모든 산출물은 프로젝트 경로 내 **`docs/`** 폴더에 저장합니다
- 파일명은 각 단계의 파일명 규칙을 준수합니다
- 파일명 형식: `{프로젝트명}-{프로세스코드}-{작업명}.md`

### 5.2 파일 저장 형식
- **마크다운 형식**으로 작성합니다
- 파일명은 **소문자**와 **하이픈(-)**을 사용합니다
- 예: `project-p01-requirements.md`, `project-p02-adr.md`

### 5.3 파일 저장 내용
- 작업 계획 수립 결과
- 작업 수행 결과
- 검증 결과
- 다음 단계로 전달할 출력 데이터

### 5.4 파일 저장 절차
1. 작업 완료 후 결과 파일 생성
2. 파일명 규칙 준수 확인
3. 파일 내용 검증
4. `docs/` 폴더에 저장
5. 저장 경로를 출력 데이터에 포함

## 6. 작업 수행 프롬프트 중요 내용

### 6.1 TASK별 작업 수행 계획 (필수 공통 작업 지침)
작업 수행 시 다음 단계를 **반드시** 수행해야 합니다:

1. **TASK 목표 확인**
   - 현재 TASK의 목표를 명확히 이해
   - 완료 기준 확인

2. **입력 데이터 확인**
   - 필요한 입력 데이터가 모두 제공되었는지 확인
   - 이전 단계의 출력 데이터 확인
   - 누락된 데이터가 있으면 요청

3. **관련 파일 확인**
   - 작업에 필요한 모든 파일을 읽고 이해
   - 기존 코드 구조 파악
   - 의존성 관계 확인

4. **작업 수행**
   - TASK 목표에 맞게 코드 작성/수정
   - 코딩 표준 준수
   - 명명 규칙 준수

5. **검증 수행**
   - 완료 기준 충족 여부 확인
   - 린터 오류 확인
   - 기능 동작 확인

6. **결과 파일 저장**
   - 작업 결과를 파일로 저장
   - 저장 경로를 출력 데이터에 포함

### 6.2 TASK별 작업 수행 처리 (필수 공통 작업 지침)
각 TASK 수행 시 다음 사항을 **반드시** 처리해야 합니다:

1. **작업 시작 전**
   - TASK 목표 및 범위 확인
   - 입력 데이터 확인
   - 관련 파일 확인

2. **작업 수행 중**
   - 단계별 진행 상황 기록
   - 문제 발생 시 즉시 보고
   - 중간 결과 저장

3. **작업 완료 후**
   - 완료 기준 충족 확인
   - 결과 파일 저장
   - 다음 TASK 준비

## 7. 코딩 표준 준수

### 7.1 언어별 코딩 표준
- **Python**: `python-coding-standards.md` 준수
- **TypeScript**: `typescript-coding-standards.md` 준수

### 7.2 프레임워크별 코딩 표준
- **FastAPI**: `fastapi-coding-standards.md` 준수
- **Next.js**: `nextjs-coding-standards.md` 준수
- **React**: `react-coding-standards.md` 준수

### 7.3 파일 명명 규칙
- **Python**: `snake_case` (예: `user_service.py`)
- **TypeScript**: `kebab-case` 또는 `PascalCase` (예: `user-service.ts`, `UserService.ts`)
- **마크다운**: `kebab-case` (예: `project-p01-requirements.md`)

## 8. 예외 상황 처리

### 8.1 복잡한 작업의 경우
- 큰 작업은 **반드시 작은 TASK로 분해** 후 진행
- 각 TASK는 **최대 1-2개 파일** 수정 범위로 제한

### 8.2 긴급 수정이 필요한 경우
- 긴급 수정도 TASK로 정의하고 진행
- 완료 후 반드시 검증 수행
